# Brain resolution — dual-mesh architecture

Two fsaverage pial-surface meshes ship to the browser, chosen per page:

| Mesh | Vertices | File | Where it renders |
|------|---------:|------|------------------|
| `fsaverage5_pial.glb` | 20,484 | `frontend/public/meshes/` | Interactive rooms (Mirror, Music, Cross-Cultural) — vertex-aligned with TRIBE's 20484-sample prediction. |
| `fsaverage6_pial_web.glb` | 81,924 | `frontend/public/meshes/` | Hero cinematic moments (Home Shot 1 + 5, About open + close) — higher polygon density catches finer gyrus/sulcus shadow detail. |

Both meshes are generated by `backend/scripts/prepare_mesh.py` via
`nilearn.datasets.fetch_surf_fsaverage`. Same brain, two resolutions.
Identical lighting + shader + material — only the geometry swaps.

## How the swap is driven

`useBrainStageStore.meshResolution` is the one source of truth. ScrollScene
writes it into the store when a section enters the viewport, alongside
position / scale / rotation / activations / lighting:

```ts
<ScrollScene
  meshResolution="fsaverage6"
  brain={{ position: [0, 0, 0], scale: 1 }}
  lighting="cinematic"
>
```

`BrainAnatomy` mounts a fresh `<MeshForResolution>` whenever the store's
`meshResolution` flips (the `key={meshRes}` prop forces remount). The
new mesh fades in via an `opacity` lerp inside the material so there's
no flash.

## How predictions reach the mesh

`backend/scripts/prepare_mesh.py` writes two sibling files:

1. **`vertex_to_region.json`** — fsaverage5 vertex index → one of our 20
   named region ids (or omitted if the vertex isn't in any tracked region).
   Currently driven by an approximate vertex-range heuristic; swap for a
   real Destrieux/Glasser parcellation in a follow-up polish pass.

2. **`vertex_correspondence_fsavg5_to_fsavg6.json`** — for each fsaverage6
   vertex, either its direct fsaverage5 counterpart (35,855 of the 81,924)
   or three nearest fsaverage5 neighbors with barycentric-ish weights
   (the remaining 46,069). This lets the fsaverage6 mesh be coloured
   from the same 20-region activation map without re-predicting.

The frontend `BrainAnatomy` fetches `vertex_to_region.json` (244 KB) on
mount and `vertex_correspondence_fsavg5_to_fsavg6.json` (5.5 MB raw, ~500
KB gzipped at the edge) only when the hero mesh is first requested.

## The honesty note

> **Predictions are interpolated on fsaverage6, not re-predicted.** TRIBE's
> output remains at fsaverage5's 20,484 cortical samples. What looks like
> finer activation detail on the hero mesh is barycentric blending of those
> same samples across a denser surface — pixels gained for aesthetic
> clarity, not bits gained for inference fidelity.

Surface this language in the About page's "What it does not do" section
when polishing.

## Regenerating the meshes

```powershell
cd backend
.venv\Scripts\Activate.ps1   # if you keep a venv
pip install nilearn trimesh scipy
python scripts/prepare_mesh.py
```

The script will:
1. fetch fsaverage5 + fsaverage6 from nilearn's data cache (~10 MB
   download the first time).
2. combine each pair of hemispheres with a 5 mm midline gap and normalize
   to ~1.6 Three.js units.
3. compute smooth-shading vertex normals.
4. write both GLBs + both vertex JSONs into `frontend/public/`.

Decimation: if you ever need a smaller fsaverage6 (current target is
60-80k verts; we ship 81,924 native), install `fast-simplification`:
`pip install fast-simplification` and re-run.

## Path 2 — fsaverage6 everywhere

If you decide the higher-resolution mesh is worth the cost in the
interactive rooms too, change the default in
`store/useBrainStageStore.ts`:

```ts
meshResolution: "fsaverage6"
```

and remove the explicit `meshResolution: "fsaverage5"` (or unset
`meshResolution` entirely) from the room pages' ScrollScenes. The
network cost on first load is +3 MB; subsequent navigations are
cached.

## Reduced-motion fallback (not yet implemented)

The spec's reduced-motion path (a static `hero-brain-static.png` that
loads instead of the mesh on `prefers-reduced-motion: reduce`) is
unimplemented. To regenerate the PNG when ready:

```powershell
pnpm dlx playwright screenshot http://localhost:3100/ \
  --viewport-size=2400,1200 \
  --clip="600,300,1200,800" \
  frontend/public/images/hero-brain-static.png
```

Then in `BrainAnatomy.tsx` short-circuit on reduced motion: if the
media query matches, render an `<img>` instead of a `<MeshForResolution>`.

## Mobile

No mobile-specific fallback yet. On a phone with a slow connection,
the 4 MB fsaverage6 GLB will take 5–10 s to load. Acceptable for the
hero shot; the page is responsive during the load thanks to the
SmoothScroll + CSS keyframe stagger that doesn't depend on the canvas.

If mobile performance becomes a problem, drop to fsaverage5 with a
`useIsMobile()` hook (write it; doesn't exist yet) that conditionally
sets `meshResolution: "fsaverage5"` on the hero scenes.
